# 这个 6.6 k star 的仓库，我差点删库了。

> 日期：2026-06-02


关注我的老粉应该都比较清楚，我在古法编程时期写过很多成体系的技术文章。有些读者朋友通过这些文章拿到了大厂 offer，有些读者朋友通过参考这些文章考上了研究生，其中不乏北大，国内顶级 top 院校，很欣慰能通过这些文章帮助到大家。

我的这些技术文章都索引在了自己的 Github 上，有幸获得大家认可，如今这个 Github 也有 6.6 K 的 star 了，算是`结丹初期`了。

[https://github.com/crisxuan/bestjavaer](https://github.com/crisxuan/bestjavaer)

但这都是之前的事情了，我现在把这些事情拎出来看，其实没什么意思。

我想说的是，这个 Github 我其实维护的不够好，因为后期教程停滞了，当偶尔看到新的 issue 和 pr 进来的时候，还是挺惭愧的。

如今 AI 时代来了，大家很多情况下已经不会再读这些硬核文章了，尤其是对于已经参加工作的各位来说，从短期来看 roi 比很低。

我如今在看到这个 Github 的时候，满眼都是自己小登时期淦文的影子。

是继续停滞更新，任由它继续像“破窗效应”一样烂下去，还是硬着头皮把它重新连载和更新上？

先不着急回答这个问题。

我先想到了`海贼王`，1997 年开始更新，到现在连载了 29 年，可以说这个作品就代表着`尾田`，尾田也代表着这个作品，纵观海贼王的更新以来，不乏有辉煌时刻和低谷时刻，尾田也因为各种事情停更过，但不管什么原因，他仍然在画，他仍然在向着他心中的那个 one piece 来 build and make 。

这么一想，其实答案就有了。

尾田没有因为时代变了、读者有了更多娱乐方式就把笔停下；那我因为“AI 时代来了”，就把自己曾经一砖一瓦垒起来的东西丢在那儿任其腐朽，好像也说不过去。

我决定不让它继续烂下去。

我想换一种方式，让这个仓库重新活过来——就像路飞从最初的木桶里跳出来，有了新的伙伴和航线。

古法编程的那些硬核长文，依然会保留在那里作为旧时代的残党。

> 而新时代，我倒想看一下，到底有没有载我的船。

我不再追求教科书式的体系搭建，而是记录一个“古典程序员”在 AI 时代的真实探索。

我希望会把他变成，一个碳基生命在硅基生命时代折腾过的印记。我的每一步都像是在做实验，有试错，有反思，有和 AI 协作时的意外发现。

这样，每当我看到新的 issue 和 pr 时，不再是惭愧，而是又能像小登时期那样，骂骂咧咧又兴奋地淦起来了。

这或许就是我的 one piece——不是某个终点，而是那个还能 build，还能 make 的状态本身。

而这个 GitHub，我也不想让其只困在结丹初期了，虽然我不像韩立一样拥有小绿瓶，但我也应该也像韩立一样，找到自己的修仙之路。

## 后续的展望

**关于内容方向**

我把自己定位成一个“古典程序员在 AI 时代的探路者”。所以你不会在这里看到 prompt 大全、AI 副业速成指南之类的东西——那些自然有更擅长的人去做。我想记录的是：一个写过多年底层代码的人，是如何用 AI 重新理解系统、重新组织知识、重新定义效率的。这些文章可能会有错误的判断、会被打脸、会在半年后回头看觉得幼稚，但没关系，实验室的意义就在于**允许实验失败**。

**关于旧内容归档**

那些“古法编程”时期的硬核长文，我会一直保留。它们是我敲下的地基，也是这个仓库 6.6K star 来时的路。

**关于“作品与开源项目”**

那个我说成是 garbage 的部分，其实是这个实验室里我最期待的一块。我相信很多有价值的东西都是从 garbage 里长出来的。我给自己定的策略是：用 AI 低成本地快速出原型，然后挑其中有生命力的持续打磨。如果你哪天在这里面看到一个像样的产品，那它一定是从一堆你不曾看到的垃圾里爬出来的。

**最后**

我不知道这个实验室最终能做成什么样，也没法保证它会变成多么厉害的项目。

但至少，我不打算再让这个仓库变成一个让我打开就惭愧的地方了。

尾田还在画他的 one piece，韩立还在找他的修仙路，我这个中登，也要探索新世界的航线了。

如果你对这个仓库有任何意见，欢迎提 issue 和 pr ，或者如果你对这个仓库有哪些指导性建议，也欢迎不吝赐教。
